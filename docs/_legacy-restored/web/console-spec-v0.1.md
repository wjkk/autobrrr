版本：v0.1  
状态：参考规格  
适用范围：MVP
关联文档：

## 1. 文档目标

这份文档不讨论品牌视觉风格，而是定义控制台页面在 MVP 阶段需要承载的信息、字段、模块关系和操作流，供产品、设计、前端、后端对齐。

目标是回答四个问题：

- 用户进入控制台后先看到什么
- 当前项目状态如何被快速理解
- 用户在关键节点如何介入和恢复流程
- 页面最小实现需要哪些字段和动作

## 2. 页面定位

项目控制台是单个漫剧项目的主工作台。

OpenClaw 可以作为外部聊天入口、通知入口和轻操作入口，但所有需要精细视觉判断、版本替换、素材处理和正式发布确认的操作，仍应回到控制台完成。

它不是传统意义上的“详情页”，而是项目推进中心。页面首要任务不是展示信息齐全，而是帮助用户在最短时间内判断：

- 这个项目现在做到哪一步
- 当前是否需要我处理
- 我能立即执行什么动作
- 最近生成了什么结果

## 3. 页面设计原则

- 首屏优先级高于信息完整性
- 当前节点优先级高于历史节点
- 当前产物优先级高于基础配置
- 当前动作优先级高于解释说明
- 失败恢复路径必须明确可见
- 审核操作必须尽量短链路完成
- 日志默认摘要展示，避免压垮主界面

## 4. 页面结构总览

建议控制台采用以下信息结构，自上而下布局：

1. 顶部项目头部区
2. 当前状态与总进度区
3. 当前节点主工作区
4. 当前产物预览区
5. 快捷操作区
6. 节点时间线 / 节点列表区
7. 日志摘要区
8. 历史版本与附加信息区

如果是桌面端，建议采用双栏：

- 左侧：项目状态、当前节点、操作、节点列表
- 右侧：当前产物、日志摘要、版本信息

如果是移动端，建议改为单栏折叠结构，但仍需保证以下内容优先出现在首屏：

- 项目状态
- 当前节点
- 当前产物
- 当前动作按钮

## 5. 页面模块说明

### 5.1 顶部项目头部区

#### 目标
快速建立项目上下文，告诉用户“这是哪个项目”。

#### 展示字段
- 项目名称
- 项目状态
- 当前阶段名称
- 最近更新时间
- 执行模式（全自动 / 关键节点审核）

#### 可选字段
- 项目题材
- 风格标签
- 目标时长
- 语言

#### 支持操作
- 返回项目列表
- 编辑项目基础信息
- 导出项目

#### 交互要求
- 项目名称应固定可见
- 项目状态应使用明显状态标签
- 若项目需要人工处理，应在头部区直接提示

### 5.2 当前状态与总进度区

#### 目标
让用户在 3 秒内知道项目总体推进情况。

#### 展示字段
- 总进度百分比
- 当前节点名称
- 当前节点状态
- 已完成节点数 / 总节点数
- 是否等待人工审核
- 是否存在失败节点

#### 建议组件
- 进度条
- 状态标签
- 简要说明文案

#### 示例文案
- 正在执行：画面生成
- 等待你审核：脚本审核
- 执行失败：配音 / 字幕
- 已完成：成片已导出

### 5.3 当前节点主工作区

#### 目标
这是控制台最核心的区域，必须默认展示当前节点的关键信息。

#### 展示字段
- 当前节点名称
- 当前节点类型
- 当前节点状态
- 开始时间
- 最近一次执行时间
- 本次执行摘要
- 是否需要人工审核
- 当前节点说明

#### 内容结构建议
- 节点标题
- 节点状态摘要
- 节点输入摘要
- 节点输出摘要
- 错误摘要（若有）

#### 交互要求
- 当前节点必须在视觉上成为页面主焦点
- 若节点等待审核，则审核按钮必须直接出现在本区
- 若节点失败，则恢复操作必须直接出现在本区

### 5.4 当前产物预览区

#### 目标
让用户不用跳页就能看到当前结果，而不是只看抽象状态。

#### 按节点类型展示
- 脚本节点：展示脚本正文摘要或前若干段
- 分镜节点：展示分镜卡片列表
- 角色设定节点：展示角色卡片
- 画面生成节点：展示图片缩略图
- 配音 / 字幕节点：展示音频、字幕摘要或关联文件
- 成片导出节点：展示视频播放器或视频封面

#### 通用字段
- 当前有效版本号
- 生成时间
- 产物类型
- 产物预览链接 / 文件地址

#### 支持操作
- 查看完整产物
- 下载 / 导出
- 打开节点详情
- 切换历史版本预览

#### 设计重点
- 控制台至少要有一个“当前产物可见窗口”
- 不要求在控制台里完成复杂编辑，但必须看得到结果

### 5.5 快捷操作区

#### 目标
集中承载当前项目和当前节点的高频操作。

#### 项目级操作
- 暂停项目
- 继续项目
- 导出结果
- 返回项目列表

#### 节点级操作
- 审核通过
- 编辑后继续
- 重跑当前节点
- 回退到当前节点
- 查看节点详情

#### 状态驱动的按钮逻辑

#### 当状态为 `running`
- 可见：暂停项目
- 可见：查看节点详情
- 可见：查看日志

#### 当状态为 `awaiting_review`
- 可见：审核通过
- 可见：编辑后继续
- 可见：暂停处理
- 可见：回退重做

#### 当状态为 `failed`
- 可见：重试当前节点
- 可见：回退上一步
- 可见：查看错误详情

#### 当状态为 `completed`
- 可见：查看结果
- 可见：重新生成终稿（如适用）
- 可见：导出结果

#### 交互要求
- 主按钮数量控制在 2 到 3 个
- 危险操作如回退、重跑需二次确认
- 不同状态下禁止展示无意义按钮

### 5.6 节点时间线 / 节点列表区

#### 目标
帮助用户理解整个项目流程，并定位上下游节点。

#### 展示字段
- 节点顺序
- 节点名称
- 节点状态
- 最近更新时间
- 是否可点击查看详情
- 是否为当前节点

#### 状态样式建议
- pending：灰色
- running：蓝色
- awaiting_review：橙色
- completed：绿色
- failed：红色
- paused：黄色或中性高亮

#### 支持操作
- 点击节点进入节点详情
- 对可恢复节点执行重跑或回退入口跳转

#### 设计重点
- 当前节点必须高亮
- 失败节点和待审核节点必须有明显提醒
- 时间线是流程感组件，不只是列表

### 5.7 日志摘要区

#### 目标
在不打断主流程的前提下，让用户理解最近发生了什么。

#### 展示字段
