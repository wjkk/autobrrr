# Web 设计 Token 与组件规格

版本：v0.1  
状态：实现规格  
适用范围：MVP
关联文档：
- `docs/web/web-route-and-page-spec-v0.1.md`
- `docs/architecture/system-architecture-role-spec-v0.1.md`

## 1. 文档目标

本文档用于约束 MVP Web 的视觉基础和组件规格，避免实现阶段退化成普通后台界面。

## 2. 视觉方向

目标风格：

- 轻背景
- 高密度信息布局
- 中性冷灰体系
- 细边框和明确层级
- 画面预览优先
- Hover 与 selected 反馈克制但明确

不建议：

- 大面积纯白无层次
- 强烈品牌色堆满全页
- 后台表格风主导
- 过度圆润的卡片堆叠

## 3. 基础 Token

### 3.1 色彩

建议变量：

- `--bg-app: #f5f6f8`
- `--bg-panel: #ffffff`
- `--bg-subtle: #f0f2f5`
- `--border-default: #e3e7ee`
- `--border-strong: #cfd6e3`
- `--text-primary: #111827`
- `--text-secondary: #5b6472`
- `--text-tertiary: #8993a4`
- `--accent-primary: #1f6feb`
- `--accent-soft: #e7f0ff`
- `--success: #0e9f6e`
- `--warning: #d97706`
- `--danger: #dc2626`

### 3.2 圆角

建议：

- 面板：`12px`
- 小卡片：`10px`
- 按钮：`10px`
- 缩略图：`12px`

### 3.3 阴影

建议只保留两级：

- 面板阴影：很轻
- Hover 阴影：略强于默认

不要做多级重阴影。

### 3.4 间距

建议基准：`4px`

常用步长：

- `8`
- `12`
- `16`
- `20`
- `24`
- `32`

### 3.5 字体与字号

建议：

- 标题：`18 / 20 / 24`
- 正文：`14 / 16`
- 辅助：`12 / 13`

原则：

- 同屏信息密度高，但不能显得拥挤
- 不要使用过大的默认后台字号

## 4. 组件规格

### 4.1 项目头部条

包含：

- 项目标题
- 状态 tag
- 当前节点
- 总进度
- 主操作按钮

要求：

- 固定在工作台顶部
- 高度稳定，不因状态变化跳动

### 4.2 节点时间线

包含：

- 节点名称
- 状态点
- 当前节点高亮
- 可点击进入详情

要求：

- 当前节点必须一眼可见
- `awaiting_review` 与 `failed` 要明显区分

### 4.3 镜头卡

必须包含：

- 缩略图
- 序号
- 标题
- 状态标识
- Hover 动作区

状态：

- `default`
- `hover`
- `selected`
- `generating`
- `failed`
- `success`

要求：

- `hover` 时出现动作按钮
- `selected` 有明显边框或底色变化
- `failed` 不只靠文字，还要有视觉标识

### 4.4 版本缩略图

包含：

- 预览图
- 版本号
- 模型标签
- 当前激活标记

要求：

- Active 与 Pending Apply 的视觉差异要明显
- Hover 时显示“激活 / 下载 / 查看详情”操作

### 4.5 日志摘要条目

包含：

- 时间
- 事件类型
- 摘要
- 状态色点

要求：

- 默认只显示摘要，不展示原始堆栈
- 错误日志可点击展开详细面板

### 4.6 动作按钮

建议层级：

- `Primary`：审核通过、开始生成、发布
- `Secondary`：查看详情、打开版本
- `Ghost`：下载、复制、查看更多
- `Danger`：回退、删除引用素材

## 5. 动效原则

建议：

- Hover 动效 `120ms - 160ms`
- 抽屉 / 弹窗 `180ms - 220ms`
- 不做大面积花哨入场
- 重点做局部反馈

必须有动画的地方：

- Hover 动作出现
- 选中态切换
- 抽屉 / 弹窗打开关闭
- 局部 loading skeleton

## 6. 响应式策略

桌面端优先设计。

移动端保留：

- 项目状态
- 当前节点
- 当前镜头预览
- 主操作按钮

折叠：

- 详细日志
- 次级版本列表
- 非关键过滤器
