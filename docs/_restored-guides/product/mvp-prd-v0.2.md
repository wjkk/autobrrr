## 1. 项目背景

当前 AI 内容创作工具大多聚焦于单点能力，例如脚本生成、角色设定、图像生成、配音生成、视频合成等。虽然这些能力不断增强，但用户在实际创作过程中仍需要承担大量流程管理工作，包括串联多个工具、维护上下游输入输出、监控生成进度、处理失败重试、记录版本差异，以及在关键节点进行人工判断。

对于漫剧这类链路较长、环节依赖明显、返工成本较高的内容形式来说，单点工具并没有真正降低生产复杂度。用户获得的是更多“能力模块”，却没有获得一个真正能持续推进项目的“执行系统”。

因此，本项目的核心机会在于：不是继续做一个更强的单点创作工具，而是做一个以 Seko 风格 Web Studio 为正式生产前台、以 OpenClaw 为协作入口和可插拔执行代理的漫剧生产托管平台。系统负责持续推进流程，用户只在关键节点做判断、修改或接管，从而显著降低生产协作成本，提高项目推进效率。

## 2. 产品定位

OpenClaw 漫剧托管平台是一个面向漫剧生产场景的 Web Studio + Agent 协作托管系统。

它的核心定位不是“辅助创作工具集”，而是“可持续执行的项目生产系统”：

- 用户创建漫剧项目并设定基础参数
- 系统围绕 Explore -> Planner -> Creation -> Publish 主流程推进项目
- 用户可在关键节点查看进度、审核结果、手动修改、重跑或回退
- 系统提供清晰的状态、日志、产物和版本记录
- 整体目标是让用户以更低的人力成本管理一个或多个漫剧项目

### 2.1 项目正式定义

本项目的正式定义如下：

- 本项目是一个以 Seko 为体验基线的开源创作系统
- 目标不是照搬其闭源实现，而是复刻其核心产品能力、主流程、状态机与交互体验
- 正式生产前台是 Web Studio
- OpenClaw 是这套系统的自然语言操作入口、协作入口和通知入口

### 2.2 目标还原度

项目当前目标不是“做一个受 Seko 启发的类似产品”，而是达到 Seko 约 90% 的核心功能与体验。

当前约定的 90% 主要按以下四个维度衡量：

1. 功能覆盖率
- Explore / Planner / Creation / Publish 主流程
- 多 Agent 策划
- 单镜与批量生成
- 模型切换
- 重试 / 替换 / 重置
- 素材提交与历史素材应用
- 导出与发布
- 多剧集与风格继承

2. 交互覆盖率
- hover reveal
- selected / pending / failed / generating 状态
- 弹窗链路
- storyboard boot 过渡层
- 版本轨交互
- 时间轴交互
- 表单阻断与 toast 反馈

3. 页面体验还原度
- 页面布局结构
- 信息密度
- 操作分区
- 工作区切换方式
- 主流程节奏

4. 状态机覆盖率
- Planner 的 `idle / updating / ready`
- Shot 的 `pending / queued / generating / success / failed`
- Version 的 `pending_apply / active / archived`
- Publish 的 `draft / submit / success / failed`

### 2.3 基线来源与核验方式

当前项目统一采用以下基线来源：

- 本地静态与复刻原型：`prototype/seko-mangju-static`、`prototype/seko-clone`
- 真实 Seko 页面与真实流程截图
- 必要时使用浏览器登录真实 Seko 进行逐页、逐弹窗、逐流程核验

核验原则：

- 本地原型用于沉淀结构化基线
- 真实 Seko 用于补齐 UI、交互、状态变化和细节偏差
- 允许在已授权登录态下使用账号与已有积分进行真实链路核验
- 禁止触发真实付费购买行为

### 2.4 参考与合规边界

- Seko 只作为产品基线和体验参考，不作为可直接搬运的源码与素材来源
- 不直接使用 Seko 的前端源码、构建产物、私有资源、品牌素材和大段原始文案
- 正式交付物中的代码、样式、图标、图片、视频、音频和 mock 数据都必须由我们自研、替换或使用合法来源
- 浏览器登录核验仅用于功能与交互研究，不触发真实付费行为
- 详细约束见 `docs/product/reference-compliance-boundary-v0.1.md`

## 3. 产品目标

MVP 阶段的核心目标如下：

- 以开源方式实现一套可稳定复刻 Seko 主流程的产品底座

- 支持用户创建并启动一个漫剧项目
- 支持系统围绕策划、分镜生成、分镜视频生成、导出与发布推进项目执行
- 支持用户实时查看项目状态、当前节点、进度和日志
- 支持关键节点人工审核、修改后继续执行
- 支持失败节点重试、节点重跑和基本回退
- 支持最终结果导出与归档
- 验证“可托管 + 可接管 + 可观测”的产品模式是否成立

## 4. 非目标

MVP 阶段不追求以下能力：

- 最优的单点生成质量
- 开放式工作流自由编排
- 多团队复杂权限体系
- 完整商业化计费系统
- 高级时间线剪辑能力
- 海量模板/素材市场
- 精细化数据分析平台

本阶段优先级是把托管生产闭环跑通，而不是在功能层面做全量铺设。

## 5. 目标用户

### 5.1 个人创作者

希望降低漫剧制作门槛，把精力集中在题材、设定、审美和成片判断上，而不是消耗在重复执行与流程管理中。

### 5.2 小型创作工作室

具备基础生产需求，但人手有限，希望在多个项目并行时减少流程盯人和协同成本。

### 5.3 内容运营 / IP 团队

更关注内容项目的可复用、可追踪、可审核，希望形成标准化生产流程，而不是依赖零散工具链。

## 6. 用户核心痛点

- 漫剧生产链路长，多个环节依赖人工串联
- 上下游依赖强，一个环节失败会拖累整个项目
- AI 结果虽多，但项目推进仍高度依赖人工
- 返工成本高，版本记录混乱
- 缺乏统一控制台，项目状态不透明
- 同时管理多个项目的心智负担高
- 用户难以知道系统当前在做什么、为何失败、下一步怎么恢复

## 7. 核心价值主张

### 7.1 可托管

用户可以把项目交给系统持续推进，而不是亲自盯住每个环节。

### 7.2 可接管

用户可以在关键节点人工介入、修改内容，并继续后续流程。

### 7.3 可观测

用户可以清晰看到当前进度、节点状态、执行日志、产物结果和失败原因。

### 7.4 可回滚

用户可以基于节点和版本恢复到某个状态，重跑某个环节，而不是整条链路重来。

## 8. 典型使用场景

### 场景 A：全自动托管

用户输入题材、风格、时长等信息后，将项目交由系统自动推进，直到完成成片，仅在系统失败或终稿阶段查看结果。

### 场景 B：关键节点审核

用户在脚本、分镜、角色设定、终稿等关键节点进行审核，审核通过后由系统继续推进。

### 场景 C：失败恢复

系统在画面生成或配音阶段失败，用户在控制台查看失败原因后执行重试或回退。

### 场景 D：人工接管后恢复自动化

用户对脚本或角色设定结果不满意，手动修改内容后点击继续，由系统基于修改后的版本继续执行。

### 场景 E：多项目并行管理

用户同时有多个漫剧项目在运行，通过项目列表和控制台快速查看哪些项目正在执行、哪些项目需要人工处理。

## 10. 产品原则

为避免产品失焦，MVP 设计遵循以下原则：

- 系统默认自动推进，人工介入为补充，不反过来
- 所有关键状态变化都必须可见、可解释
- 用户无需理解底层执行细节，也要知道“当前在哪、出了什么问题、下一步能做什么”
- 控制台优先服务“项目推进”，不是信息堆叠
- 关键操作必须具备安全确认，避免误重跑、误回退
- 状态模型必须稳定简洁，减少实现和认知复杂度

## 11. 整体业务流程

### 11.1 主流程

1. 用户创建项目
2. 系统初始化项目配置与默认流水线
3. 系统开始执行脚本生成
4. 用户在脚本审核态进行通过或编辑后继续
5. 系统执行分镜生成与角色设定
6. 用户在 Web 工作台中查看分镜板、镜头状态和版本
7. 系统执行画面生成、配音、字幕等后续节点
8. 用户在终稿阶段确认结果并准备发布
9. 系统执行成片导出与发布
10. 用户查看结果并归档项目

### 11.2 异常流程

- 节点执行失败，系统将状态标记为 failed
- 用户查看错误原因并选择重试
- 用户选择回退到上游节点重新生成
- 用户手动修改节点输出后继续
- 用户暂停整个项目，稍后恢复执行
