## 1. 文档目标

本文档用于把 MVP 从产品方案拆到研发执行层，帮助团队明确：

- 先做什么
- 后做什么
- 每一阶段验证什么
- 哪些是必须项
- 哪些可以后置

原则是：优先验证托管系统闭环，再逐步增强生成能力。

## 2. 路线图总原则

- 先验证流程，不先卷质量
- 先验证单项目闭环，再扩多项目管理
- 先固定节点流水线，不先做自由编排
- 先保证状态、日志、审核、回滚清晰，再优化体验细节
- 先把 OpenClaw 接成协作入口和可插拔执行器，再逐步扩执行能力

## 3. 建议阶段划分

建议分为 4 个阶段推进：

- Phase 0：技术底座与最小骨架
- Phase 1：托管闭环 MVP
- Phase 2：半真实生成链路
- Phase 3：成片闭环与内测准备

## 4. Phase 0：技术底座与最小骨架

目标：搭起可持续开发的最小基础设施。

建议周期：1 周

### 必做项

- 初始化前端项目
- 初始化后端 API
- 建立 PostgreSQL + Prisma 数据模型骨架
- 接入 Redis + BullMQ
- 搭建对象存储抽象层
- 约定项目、节点、run、镜头、镜头版本、审核、事件的基础模型
- 建立 OpenClaw 协作入口与适配层占位

### 建议交付

- 项目仓库结构
- 基础环境启动脚本
- 数据表初版迁移
- 简单健康检查接口
- 一套固定节点枚举与状态枚举

### 验证标准

- 开发环境可以一键启动
- 可以创建一个空项目并初始化节点
- 可以通过后端创建一条假 run 记录

## 5. Phase 1：托管闭环 MVP

目标：不依赖完整真实生成能力，先验证“可托管 + 可接管 + 可观测”的系统闭环。

建议周期：2 周

### 核心目标

- 用户可创建项目
- 控制台可查看当前状态
- 固定流水线可推进
- 审核态可人工通过或编辑后继续
- 失败节点可重试和回退
- 日志和版本可见

### 必做项

### 前端
- 项目创建页
- 项目列表页
- 项目控制台页
- 分镜板 / 镜头工作台基础版
- 节点详情页基础版

### 后端
- 项目 CRUD
- 节点初始化
- 控制台聚合接口
- 审核接口
- 重试接口
- 回退接口
- 镜头生成与版本激活接口
- 发布草稿与发布接口
- 版本记录机制
- 事件日志机制

### OpenClaw / 执行层
- 节点执行协议第一版
- 至少支持 mock 执行
- 支持 `run.started`、`run.log`、`run.completed`、`run.failed`

### 可先 mock 的节点

- 脚本生成
- 分镜生成
- 角色设定
- 画面生成
- 配音 / 字幕
- 成片导出

即使内部不是真实调用，也要保证每个节点有明确输入输出和状态流转。

### 验证标准

- 能从创建项目一路跑到完成态
- 至少一个节点可以进入 `awaiting_review`
- 用户编辑后继续能够生成新版本并推进下游
- 镜头板能够稳定展示 `queued / generating / success / failed`
- 至少一个镜头可完成版本激活
- 某节点失败后可重试成功
- 某节点可回退后重新执行
- 控制台能够稳定显示当前节点、镜头板状态、当前产物、日志摘要

## 6. Phase 2：半真实生成链路

目标：接入部分真实生成能力，验证上下游承接与故障恢复。

建议周期：2 周

### 优先接入的真实节点

- 脚本生成
- 分镜文本生成
- 角色设定
- 图像生成

### 此阶段建议

- 视频导出先弱化
- 优先验证文本 -> 分镜 -> 图像链路
- 强化错误分类和重试逻辑
- 补充更多执行日志摘要

### 必做项

### OpenClaw / 执行层
- 接入真实 LLM provider
- 接入图像生成 provider
- 支持产物上传到对象存储
- 支持 provider 错误映射

### 后端
- 失败分类统计
- 更清晰的错误摘要生成
- 版本恢复接口完善

### 前端
- 当前产物预览增强
- 失败状态展示增强
- 镜头版本面板增强
- 历史版本列表增强

### 验证标准

- 至少 2 到 3 个节点使用真实能力执行
- 上游版本变更后，下游可以基于新版本继续执行
- 失败节点可以基于明确原因重试
- 控制台里用户能看懂系统哪里失败、如何恢复

## 7. Phase 3：成片闭环与内测准备

目标：形成完整可演示、可小范围内测的 MVP。

建议周期：2 到 3 周

### 增强方向

- 配音接入
- 字幕接入
- `FFmpeg` 导出接入
- 最终成片页补齐
- 导出记录补齐
- 内测用的基本监控和告警

### 必做项

### 前端
- 成片结果页
- 导出记录展示
- 完成态体验优化

### 后端
- 导出任务管理
- 产物归档与下载链接
- 项目完成态判断完善

### OpenClaw / 执行层
- 配音与字幕节点执行协议
- 视频导出节点衔接
- 更完整的中间产物整理策略

### 验证标准

- 一个项目可跑通至最终导出
- 至少支持一次人工审核介入后继续完成
- 至少支持一次失败恢复后完成
- 可生成内测演示样例项目

## 8. 每阶段的“不要做”

### Phase 0 不要做

- 不要在 Phase 0 陷入像素级全量还原，但必须先把 Seko 式工作台骨架做对
- 不要做自由工作流
- 不要接太多 provider

### Phase 1 不要做

- 不要过度优化生成质量
- 不要做复杂权限系统
- 不要做高级日志平台

### Phase 2 不要做

- 不要急着做完整商业化计费
- 不要急着做多团队协作
- 不要急着做复杂时间线编辑器

### Phase 3 不要做

- 不要因为想做更强的编辑器而拖延闭环上线
- 不要把系统搞成内容创作全家桶

## 9. 人员视角的任务拆分建议

如果团队很小，可按角色粗分：

### 产品 / 方案
- PRD 收敛
- 状态流转规则确认
- 审核和回退策略确认
- 节点输入输出定义

### 前端
- 项目创建与列表
- 控制台信息架构
- 分镜板 / 版本面板
- 节点详情
