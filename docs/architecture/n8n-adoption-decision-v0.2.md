# n8n 采用决策（v0.2）

版本：v0.2  
日期：2026-03-09  
状态：现行决策（对齐 Explore/Planner 架构）

关联文档：
- `docs/architecture/system-architecture-role-spec-v0.2.md`
- `docs/specs/explore-planner-backend-guidance-v0.2.md`

## 1. 结论

1. 不将 `n8n` 作为核心制作链路引擎。
2. 可采用 `n8n`，但仅限外围自动化层。
3. 核心状态机、版本生效、任务编排与审计必须在业务后端。

一句话：

`核心制作链路不用 n8n，外围自动化可以用 n8n。`

## 2. 决策依据

Explore/Planner 当前核心复杂度在于：

1. 统一提交语义（首次确认 vs rerun）
2. 细化版本历史只读
3. 步骤级进度与渐进快照
4. 局部 patch 仅作用于目标版本
5. 配置项（分镜模型/比例）持久化

这些能力都依赖强业务状态机与数据库约束，不适合由工作流平台承载真相。

## 3. 允许使用 n8n 的场景

1. 通知分发：飞书、Slack、邮件。
2. 运营自动化：日报、周报、异常汇总。
3. 外部系统同步：CMS/数据仓库/BI。
4. 非核心审批流编排。

共同特征：

1. 异步可重试
2. 失败不破坏核心业务状态
3. 可由事件订阅触发

## 4. 禁止使用 n8n 的场景

1. 项目/策划状态机推进。
2. 大纲确认与细化版本创建。
3. 版本激活与回滚语义。
4. 分镜 patch 生效与审计主链路。
5. 核心 AI 任务调度与恢复。

## 5. 推荐接入模式

1. Domain Backend 产生业务事件（如 `planner.refinement.completed`）。
2. n8n 仅订阅事件并执行外围动作。
3. n8n 若需回写系统，只能调用受控业务 API，不直写核心表。

## 6. 治理要求

1. 建立 workflow 命名规范与 owner。
2. 关键 workflow 开启失败告警。
3. workflow 输入输出做 schema 校验。
4. 将 n8n 与核心数据库写权限隔离。
